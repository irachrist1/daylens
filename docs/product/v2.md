# Daylens V2 direction

**Status:** Accepted product direction. Complete implementation drafts are indexed in the root README and tracked for review in the [to-do list](../TO-DO.md).

V2 makes the desktop application exceptionally good at understanding a person’s day before Daylens expands to the web or organizations. The durable product promise and boundaries live in [Product direction](product.md). This document defines what V2 includes, how the pieces fit together, and what must wait.

## Product sequence

The first customer is an individual who wants automatic time tracking, a reliable memory of everything they did on their computer, and an AI agent that can answer questions about it.

The work proceeds in this order:

1. Build and validate the complete desktop experience.
2. Add remote memory search and AI chat on the web immediately after the desktop experience is dependable.
3. Introduce organizational products only after the individual product has proven useful and retained people.

The desktop milestone is not complete because every planned surface exists. It is complete when capture is dependable, Timeline and Apps agree, corrections persist, search retrieves useful details, and the AI agent consistently answers real questions using the same memory.

## Existing features during the transition

The shipped application predates this direction. Every existing feature — and every AI feature in particular — has one explicit disposition so implementation never has to guess. Nothing is silently deferred.

**Removed:**

- **Focus score and derived deep-work metrics.** Computed scores about focus quality contradict the product boundaries, including where they only feed AI tool payloads.
- **Distraction alerts and distraction counting.** Daylens does not diagnose distraction.
- **The carryover nudge.** It shipped without an accepted product decision and is deleted with the brief rebuild, not migrated.
- **CLI provider modes** (`claude-cli`, `chatgpt-cli`, `gemini-cli`, `codex-cli`) are not part of Version 2. They remain personal, unmarketed development conveniences until the provider-permission questions in the to-do list are answered in writing. The chat agent already declines to run its tool loop through them.

**Rebuilt on the shared systems before Version 2 ships:**

- **The chat agent** moves onto the context packet, disclosure record, and shared corrected facts defined by the agent specifications. The single loop, grounding verification, and bench parity survive; the facts underneath change.
- **AI actions in chat** (rename and merge today) grow into the full previewed, confirmed, reversible action set the [AI agent specification](../specs/ai-agent.md) defines: rename, merge, split, category, project and client attribution, meeting relationship, exclusion, and forgetting saved memory.
- **Automatic day analysis** (the regroup-and-relabel pipeline) becomes the interpretation agent of the [agent runtime specification](../specs/agent-runtime-and-context.md): versioned inference over evidence, evaluated offline against fixtures before it writes product state, never overriding corrections.
- **Conversational and work memory** (including client-scoped facts) becomes supplied memory under the [memory specification](../specs/memory-and-entities.md): proposed explicitly, saved only on confirmation, reviewable and deletable.
- **Wrapped is rebuilt, not parked.** Day, week, month, and year wraps move onto the same corrected facts, interpretation, and voice as Timeline, Apps, and the agent, per the [Wrapped specification](../specs/wrapped.md). Wrapped is part of the Version 2 release gate.
- **Proactive briefs are rebuilt, not parked.** The morning brief, evening recap, and weekly brief stay shipped and frozen during the foundation work, then move onto the same period facts, interpretation, and voice per the [Briefs specification](../specs/briefs.md) — never a separate summarization pipeline. Their scheduling and notification delivery are kept where they work. Briefs are part of the Version 2 release gate.
- **Natural-language search** joins the hybrid retrieval defined by the memory specification: structured queries, exact full-text, then local semantic search.
- **Signal enrichment** (git and calendar context feeding narratives) folds into the canonical connector evidence system instead of remaining a parallel collection path.
- **The model picker** is rebuilt to the AI agent specification: provider, relative quality and speed, availability, and estimated allowance use, without model choice changing Daylens facts.

**Kept, migrating to shared facts with everything else:**

- **The answer-integrity layer**: grounding verification with corrective retry, deterministic time-chunk rendering, source citations, and tool traces persisted with each message.
- **AI threads** with rename, archive, deletion, and paging; **AI-generated thread titles**; **follow-up suggestions**.
- **Artifacts**: CSV, spreadsheet, and report files produced from tool results.
- **Bring-your-own-key providers** (permanent) and the **managed transport with reserve-and-settle metering** behind the billing service.
- **AI infrastructure**: rate limiting, provider circuit breaker, cancellation, usage retention, and prompt caching.
- **The local MCP server** exposing Daylens facts, and **user-configured MCP servers** in chat under the hardened environment and permission boundaries.
- **The shared voice contract** and the person's chosen tone (straight, warm, witty) across every generated surface.
- **User-initiated focus sessions** as explicit supplied context: the person declares a session and may label and reflect on it. The automated distraction counting formerly attached to them is removed.

**Replaced:**

- **Server-side web AI over plaintext Convex data** is replaced by the encrypted web companion's gateway design after the desktop milestone. The current web companion and Convex sync are frozen — no new features — and continue for already-linked people until the [web companion specification](../specs/web-companion.md) replaces them.

**Recorded non-decisions:**

- **Settings has no standalone specification.** Each feature's specification owns its own settings behavior; this is a recorded decision, not an omission.

## Product-market fit

Daylens should be evaluated against the work people currently do with their own memory, browser history, calendars, time trackers, meeting tools, and general AI assistants.

The first important outcomes are:

- a person recognizes their day without manually reconstructing it
- Daylens retrieves something the person remembers seeing but cannot locate
- the AI agent answers a question that would otherwise require checking several applications
- the person returns for weekly, monthly, project, client, and meeting recall

The website work should include:

- a competitor matrix covering automatic capture, Timeline, Apps, search, memory, AI answers, meetings, connected tools, corrections, privacy, platforms, organizations, and pricing
- focused comparisons with automatic time trackers, AI memory products, and meeting-memory tools
- a landing page built around specific questions Daylens can answer and the difference between raw tracking and a memory an agent understands

Daylens should be compared with products such as Rize, Timely, Timing, Memtime, ActivityWatch, RescueTime, Littlebird, Rewind or Limitless, and Granola. Greptile and CodeRabbit are useful references for how persuasive comparison pages use direct claims and visible evidence; they are not Daylens competitors. See [Greptile versus CodeRabbit](https://www.greptile.com/greptile-vs-coderabbit) and [Rize’s automatic time-tracking comparison](https://rize.io/compare/automatic-time-tracking-software).

## How V2 should sound

Every surface should describe the activity Daylens understands before describing the telemetry used to infer it.

Daylens should say that you developed a feature, read a specific article, reviewed a financial report with a client, or worked across planning and implementation. It should name the subject, project, client, people, and outcome when the evidence supports those details. Application names, window titles, URLs, calendar records, and connector data remain inspectable evidence underneath the interpretation.

A short observation such as “Solid session there” is welcome when it is relevant and supported. Daylens should feel observant, not chatty for its own sake, and it should never turn incomplete evidence into a judgment about focus or productivity.

When the record conflicts, the answer should name the actual uncertainty in one natural sentence. “I found the meeting on your calendar, but no matching device or Granola activity” is useful. “Based on the available metadata” is not.

## Capture and evidence

Daylens captures work, entertainment, shopping, and general browsing across the device by default. Personal activity belongs in the same memory and Timeline, with filters, application and website exclusions, temporary pause controls, and clear deletion controls.

Private or incognito windows are never captured. Exclusions and pauses are enforced before evidence is persisted or sent downstream.

V2 uses three evidence levels:

| Level                     | Evidence                                                                                                                                                        | Default                                        |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------- |
| Core desktop evidence     | Foreground application, window title, process and application identity, active browser page and URL, start and end times, idle, lock, sleep, and capture health | On after onboarding consent                    |
| Connected evidence        | Calendars, meetings, GitHub, Linear, Granola, communication tools, documents, and other sources that add facts the desktop cannot infer reliably                | Explicit connection, read-only first           |
| High-sensitivity evidence | Screen frames, extracted screen text, accessibility data, transcripts, and similar content                                                                      | Separate opt-in, local-first, narrow retention |

Every capture and connector adapter should produce one canonical evidence envelope containing:

- source and source record identifier
- observed time or time range
- application, page, file, meeting, person, project, or client references
- raw and normalized labels
- sensitivity, confidence, and provenance
- permission and exclusion state
- schema version

Timeline, Apps, search, memory, and the AI agent consume this shared evidence model rather than inventing capture rules independently.

## Screen-context experiment

Screen context is a prototype, not a committed default feature. It should be tested only after metadata and connector evidence are working well enough to show which questions still lack context.

The prototype uses event-driven or bounded periodic frame sampling. It does not continuously record video.

The prototype must:

1. Require separate consent and show a persistent capture state.
2. Respect exclusions, private windows, password surfaces, and pauses before capture.
3. Extract useful text and structured details locally.
4. Atomically persist the derived evidence, provenance, and search index before marking the frame safe to delete.
5. Delete the raw frame after successful extraction, targeting no more than 24 hours of retention.
6. Quarantine and retry failed extractions instead of silently deleting the only copy.
7. Notify the person about a failed backlog and stop unbounded accumulation.
8. Provide immediate pause and deletion controls.

PostHog may receive experiment measurements such as capture attempts, extraction success, latency, completeness, correction rate, and downstream answer usefulness. It must never receive screenshots, extracted text, titles, URLs, names, or other captured content.

Daylens does not record meeting audio in V2. Meeting notes and transcripts come from explicit integrations such as Granola.

macOS uses ScreenCaptureKit and requires system permission. Windows uses `Windows.Graphics.Capture` and its system-visible capture controls. See [Apple ScreenCaptureKit](https://developer.apple.com/documentation/ScreenCaptureKit) and [Windows screen capture](https://learn.microsoft.com/en-us/windows/apps/develop/media-authoring-processing/screen-capture).

## Connectors

Sources that form canonical Daylens memory use direct adapters. A wrapped integration platform may provide long-tail sources without making the core memory dependent on one vendor.

The connector order is:

1. Google Calendar and Outlook Calendar
2. GitHub, Linear, and Granola
3. Slack, Microsoft Teams, Gmail, and Outlook email
4. Optional document, customer, and long-tail tools based on demonstrated demand

Google Calendar and Outlook Calendar begin with narrow read-only scopes. GitHub and Linear begin with direct read adapters. Granola uses its MCP connection for individuals and its API where workspace access permits it. See the [Google Calendar API](https://developers.google.com/workspace/calendar/api/v3/reference), [Microsoft Graph calendar API](https://learn.microsoft.com/en-us/graph/api/resources/calendar-overview?view=graph-rest-1.0), [GitHub REST API](https://docs.github.com/en/rest), [Linear GraphQL API](https://linear.app/developers/graphql), [Granola MCP](https://docs.granola.ai/help-center/sharing/integrations/mcp), and [Granola API](https://docs.granola.ai/introduction).

Composio may be used behind a Daylens-owned adapter for optional long-tail connections. Daylens owns the canonical connector interface, pins tool versions, and prevents vendor credentials, scopes, and availability from leaking into the memory model. See [Composio authentication](https://docs.composio.dev/docs/authentication) and [Composio toolkits](https://docs.composio.dev/reference/v3/api-reference/toolkits).

Connections can be managed in Settings. The AI agent may also request a specific connection when it can explain exactly how that source would improve the current answer.

All connectors remain read-only in V2. External actions wait for a separately accepted workflow.

## Timeline

The first V2 Timeline release perfects one calendar-like day view. Week and month Timeline views follow only after the daily reconstruction is correct.

A block represents one contiguous stretch of activity with a coherent subject or goal. It is not one application session and is not created merely because a fixed amount of time passed.

- Strong boundaries include idle or sleep, a meeting edge, a sustained subject change, and an explicit correction.
- Brief application and page switches remain evidence inside the surrounding block.
- Meetings appear as unified Timeline blocks enriched by calendar, transcript, and surrounding device evidence.
- A scheduled event becomes “you met” only when the available evidence supports that interpretation.
- Each block has a specific name, time range, duration, project or client when known, and expandable evidence.
- Renaming, merging, splitting, excluding, changing attribution, and undoing a correction are first-class commands.
- Corrections survive every rebuild and change every downstream surface.
- Personal and entertainment activity appears in the same Timeline with filters and exclusions.

Timeline owns no private version of active time. Apps, search, the AI agent, MCP, and sync consume the same canonical facts and totals.

## Apps

Apps explains what happened inside each application. It is not a leaderboard of application totals.

The first V2 release supports day, week, and month ranges. Each application view connects time with pages, files, meetings, projects, clients, related Timeline blocks, and useful questions the AI agent can answer.

A browser is a container. Websites and pages are explained inside the browser and are not counted again as additive time.

Projects and clients are reusable entities and filters across Timeline, Apps, search, and the AI agent. They do not become top-level tabs in the first V2 release.

Apps and Timeline use the same canonical activity queries and must reconcile exactly under the same filters.

## Search and memory

Daylens uses hybrid retrieval rather than placing an entire history into one model prompt.

- Structured queries answer questions about time, duration, applications, projects, clients, people, meetings, and known entities.
- Local full-text search finds exact titles, URLs, filenames, meeting names, and quoted phrases.
- Local semantic retrieval handles vague memories such as “the TV page with the best discount.”
- Ranking uses time, entity relationships, source quality, corrections, and semantic relevance.
- The entity graph connects people, applications, pages, files, meetings, repositories, projects, clients, and Timeline blocks.
- Observed, connected, explicitly supplied, and inferred memory remain distinguishable.
- Exact provenance and confidence remain internal facts with expandable supporting evidence.

Semantic embeddings are generated locally by default. Remote embeddings require an explicit choice and send only the minimum permitted text.

The AI agent may propose saving a useful fact or preference learned during conversation. It becomes durable memory only after the person confirms it.

Core activity and derived memory remain until the person deletes them. After one year, Daylens recommends an export. Partitioning, indexing, compaction, and cold access keep long histories fast without making deletion a performance requirement.

Deleting a source removes it from exact search, semantic indexes, memory, and downstream surfaces.

## AI agent

The AI agent is the conversational interface to the same memory used by Timeline and Apps. It resolves likely dates and entities, narrows the evidence, calculates facts deterministically, and then uses a model to explain them.

The initial tool layer supports:

1. Resolving natural-language time ranges.
2. Resolving people, applications, pages, files, meetings, projects, clients, and repositories.
3. Retrieving Timeline blocks for a period and filter.
4. Aggregating canonical time by entity or category.
5. Searching exact and semantic memory.
6. Retrieving evidence and source links.
7. Querying a connected source through a Daylens connector.
8. Comparing periods or entities.
9. Asking for clarification or explaining which missing connection would improve the answer.
10. Previewing and applying an explicit, reversible Daylens correction.

V2 actions are limited to Daylens itself: renaming, merging, splitting, excluding, correcting attribution, and organizing memory. Sending messages, editing calendars, and changing external systems are outside V2.

Correct question answering comes before proactive delivery: briefs are rebuilt in the final wave, after the agent answers accepted fixtures correctly. Summaries, Timeline descriptions, wraps, and briefs are all produced from the same evidence and interpretation system rather than becoming separate AI features.

People choose the managed model they want to use. The interface explains practical quality, speed, and usage differences without allowing model choice to change the meaning of Daylens facts.

Bring-your-own-key remains permanently available. BYOK requests remain available when the managed allowance is exhausted because they do not consume Daylens model credit.

## Privacy, retention, and sync

- Local activity stays local by default.
- Permissions appear when a feature needs them and explain the benefit in product language.
- Application, website, private-window, and pause rules are enforced before persistence.
- Screen content, transcripts, and accessibility data remain separate high-sensitivity categories.
- Model requests receive only the evidence required for the current question after exclusions and redaction.
- Every AI answer shows source and privacy indicators, and the exact evidence sent to the model is inspectable on demand.
- Export and deletion cover raw evidence, derived facts, embeddings, corrections, connected data, and remote copies.

Encrypted cross-device sync contains organized facts, entities, corrections, summaries, and AI threads. Raw desktop evidence and raw screen frames remain on the source device.

## Billing and availability

Core local capture, Timeline, Apps, corrections, and non-AI memory remain useful without payment or a model provider.

Managed AI is paid and metered through the billing service. Cloud-backed features share the managed entitlement. When the allowance is exhausted, managed AI and cloud-backed features pause while the local product and BYOK AI continue to work.

The initial trial lasts seven days, requires no payment card, and includes a fixed managed-AI allowance. Annual billing receives a 20% discount.

The permanent monthly price and included allowance are chosen after desktop beta, representative question benchmarks, provider-cost measurement, and willingness-to-pay testing. `$14.99/month` may be used for internal planning but is not a committed public price.

Usage is shown in understandable units and money rather than tokens alone. Managed calls reserve credit before execution and settle actual cost afterward. A billing or provider outage must not affect local capture or access to stored memory. BYOK calls remain outside managed billing.

## Web companion and organizations

The web companion begins immediately after the desktop milestone is dependable. Its first job is remote memory search and AI chat, not duplicating every desktop setting and surface.

AI threads continue between desktop and web by default. Sync sends organized, privacy-filtered facts and the references needed by those threads; it does not send raw desktop capture.

Organizational work begins only after the individual product succeeds. The first sharing primitive is a project, client, time, or status summary that the individual has reviewed and approved. Organizations do not receive passive access to personal evidence, monitoring screens, or productivity rankings.

## Implementation waves

The waves are implementation order, not releases. No wave is shipped or called Version 2 on its own; Wave 1 in particular is internal foundation work. Version 2 ships only when the [release gate](#version-2-release-gate) below is satisfied, and it takes as long as it takes.

Specifications are reviewed, accepted, and implemented in dependency order. Within a wave, listed items can proceed in parallel; a wave begins when the previous wave's blocking items are accepted.

**Wave 0 — repairs and in-flight foundations.** The agent machine-tool privacy hardening and the capture evidence-contract foundation slice. Both are complete and accepted: machine tools are scoped to visible home paths with minimal child environments, and focus events validate through one typed contract behind a dedicated repository.

**Wave 1 — one canonical fact path.** Accept [Capture and evidence](../specs/capture-and-evidence.md), [Timeline](../specs/timeline.md), [Apps](../specs/apps.md), and [Onboarding and consent](../specs/onboarding-and-consent.md).

1. Capture migration slices, in the order the capture specification defines.
2. The shared corrected activity-fact query — its own ticket; it is the seam every surface depends on.
3. Timeline day reads behind the shared query (after 2).
4. Apps totals through the same query with reconciliation tests (after 2, parallel with 3).
5. The onboarding consent gate (parallel; small and privacy-critical).
6. In parallel, the verification scaffolding: representative-day fixtures extending the existing `tests/timeline-eval` fixture format, an upgraded-representative-database migration fixture, and a minimal running-app smoke harness that screenshots the primary routes over a seeded day.

**Wave 2 — memory and the agent.** Accept [Memory and entities](../specs/memory-and-entities.md), [AI agent](../specs/ai-agent.md), and [Agent runtime and context](../specs/agent-runtime-and-context.md).

1. Entity and memory repositories with exact retrieval.
2. Agent read tools on shared corrected facts, with accepted answer fixtures.
3. Context packet, disclosure record, and context inspector (assembled and inspected without a model call first).
4. Local semantic indexing, after the local-embedding research in the to-do list concludes.

**Wave 3 — connectors, billing, privacy boundaries.** Accept [Connectors](../specs/connectors.md), [Billing and entitlements](../specs/billing-and-entitlements.md), and [Privacy, retention, and sync](../specs/privacy-retention-and-sync.md).

1. Connector contract with a fake provider, then Google Calendar.
2. Entitlement-snapshot consolidation on the desktop.
3. Deletion-by-ownership centralization and the sync allowlist contract tests (the local halves of the privacy specification).
4. The [screen-context experiment](../specs/screen-context.md) can start any time after wave 1 once its specification is accepted; it touches no shared seam.

**Wave 4 — the complete agent, Wrapped, and briefs.** Accept the [Wrapped](../specs/wrapped.md) and [Briefs](../specs/briefs.md) specifications.

1. The full Daylens action set with preview, confirmation, atomic apply, and undo.
2. Human handoffs: the agent pauses for clarification, permission, connection, or confirmation and resumes from an explicit pending state.
3. The rebuilt model picker across managed models and BYOK, with allowance estimates.
4. Wrapped rebuilt on shared corrected facts, interpretation, and voice, reconciling with Timeline and Apps.
5. Briefs rebuilt on the same period facts and voice, the carryover nudge deleted, scheduling and delivery kept where they work.
6. Runtime evaluation: the accepted fixture suites and the private real-day benchmark pass, including the reviewed day's wrap and briefs.
7. Interpretation-agent rollout: automatic day analysis replaced by versioned inference that never overrides corrections.

**After Version 2.** The [web companion](../specs/web-companion.md) waits for the release gate and the browser-encryption feasibility research. [Organizational sharing](../specs/organization-sharing.md) is deferred entirely.

Every implementation ticket, in every wave, ships through the same gates: focused regression tests, type checking, the offline suite, contract checks for touched boundaries, privacy-boundary tests when capture, AI, export, or sync behavior changes, verification in the running application on affected platforms, and explicit acceptance before the ticket is deleted.

## Version 2 release gate

Version 2 is one release, and it does not ship until every requirement below is implemented, tested against real days, and accepted in the running product:

- Capture is dependable on supported platforms, and Timeline, Apps, search, memory, MCP, wraps, and the agent read one shared source of corrected facts with reconciling totals.
- The complete AI agent works end to end: memory and search over projects, clients, people, meetings, pages, files, and repositories; deterministic context assembly with recorded disclosure; evidence and source inspection on every answer; permitted file access under the three-state model; scoped tools; the priority read-only connectors; model selection across managed models and BYOK; human handoffs; and the safe, reversible Daylens action set.
- The agent's answers are truthful: they agree with Timeline and Apps, name real conflicts instead of asserting agreement, and pass the accepted question fixtures and the private real-day benchmark.
- Wrapped is rebuilt on the shared facts, interpretation, and voice, and its totals reconcile with Timeline and Apps.
- The morning brief, evening recap, and weekly brief are rebuilt on the same period facts and voice, never disagree with the surfaces they open, and pass the real-day check; the carryover nudge is removed.
- Managed billing, BYOK, privacy boundaries, exclusions, deletion, and export work at their real service and storage boundaries.
- The previously failing private real day is reviewed and accepted because the reconstruction is genuinely correct and useful.

Internal foundation waves are never shipped or announced as Version 2. Anything listed as removed, replaced, or decision-pending in the disposition list is resolved — not quietly missing — before release.

## Desktop milestone

These are the same conditions the release gate requires, restated as the entry test for web work. The desktop is ready to move into the web milestone when:

- representative days are captured reliably across supported desktop platforms
- Timeline reconstructs a recognizable day and Apps reconciles with it
- meetings, projects, clients, pages, and files are connected when evidence supports them
- corrections are reversible, durable, and reflected everywhere
- exact and semantic retrieval find representative memories
- the AI agent answers accepted question fixtures specifically and accurately without leading with telemetry
- evidence and model context remain inspectable
- exclusions, deletion, retention, and managed usage limits work at their actual storage and service boundaries
- product analytics measure reliability and usefulness without receiving private captured content

The individual web milestone begins only after these conditions are demonstrated in the running desktop application.
