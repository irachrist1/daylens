# Daylens v2 — the plan

This file summarizes the plan. Detailed specs live in `docs/specs/`. The feature
registry at `docs/plans/FEATURE-REGISTRY.md` tracks what's broken, what's unverified,
and what's missing — with screenshot evidence.

## What's broken

Every visible problem traces back to one root cause: **the app groups activity by
which app was open, not by what you were doing.** That decision produced 53 blocks
for a day that had about 8 real activities. Everything downstream — titles,
categories, totals, recaps, AI answers — inherited the bad data.

Concretely:

1. **Blocks are named wrong.** "Ubiquiti Account", "UOS Server", "Cross-platform
   activity tracker", raw YouTube video titles. These are window titles, not
   descriptions of work.

2. **Same work split into duplicate blocks.** Two "UOS Server" blocks back-to-back
   for the same activity.

3. **Wrong categories.** A coding block tagged ENTERTAINMENT because Netflix was
   open for 42 seconds. Network work tagged SOCIAL because one X.com tab was open.

4. **Numbers disagree with each other.** The week card says 42h 2m, the review text
   says 46h 19m — on the same screen.

5. **The AI can't answer the main question.** "What did I work on today?" gets an
   apology asking you to paste data.

6. **Grades nobody asked for.** Score 74, Focused 8h 3m, Drift 5h 56m. Daylens
   shows your day — it doesn't grade it.

7. **System noise counted as activity.** `loginwindow` shows as 16–50 hours of
   "app usage."

8. **Leisure mixed into work highlights.** Netflix and X appear in "What mattered."

9. **Morning brief is a slideshow.** Should be one screen: what you left open, pick
   it up.

10. **Views don't talk to each other.** The Timeline, Apps, AI, and weekly review
    each compute their own numbers instead of reading from the same source.

See `docs/plans/screenshots/` for evidence of each one.

## The rules (behavioral invariants) for the app to always follow

These are laws the app must always follow. They're not features — they're the
physics of Daylens. Every agent reads these before building anything.

1. **One block = one stretch of one intent.** A block is defined by what you were
   *doing*, not which app was in focus. Apps are evidence inside a block, not
   boundaries between blocks.

2. **Brief detours are absorbed.** A 2-minute X.com glance during 3 hours of
   network work folds into the network block. It doesn't create a SOCIAL block.
   Proposed cutoff: under 10 minutes = absorbed; 10+ minutes = its own block.

3. **Same-intent neighbors merge.** Two adjacent blocks with the same purpose
   become one block. The two "UOS Server" blocks become one "Configuring the
   home network" block.

4. **Block height = duration.** A 3-hour block is 3x the height of a 1-hour block.
   Always. No exceptions.

5. **No block is named after a window title.** Never "Ubiquiti Account", never
   "Cross-platform activity tracker", never a YouTube video title. The name says
   what you were *doing*: "Configuring the home network."

6. **A single off-task tab never sets a block's category.** 42 seconds of Netflix
   during coding = DEVELOPMENT, not ENTERTAINMENT.

7. **One truth, three views.** Timeline, Apps, and AI all read the same block facts.
   If the Timeline says 3 hours of network work, the AI says the same. No view
   computes its own totals. The recap total always equals the sum of the blocks.

8. **Your corrections always win.** If you rename, merge, or hide a block, that
   sticks forever — across rebuilds, re-analysis, and memory updates. The app
   never silently overwrites your edit.

9. **No grades.** No Score, no Focus rating, no Drift number. Daylens shows your
   day — it doesn't judge it.

10. **When Daylens doesn't know, it says so.** It never fills a gap with a guess.
    An empty stretch says "machine off" or "untracked" — not nothing.

11. **System noise is invisible.** `loginwindow`, `UserNotificationCenter`, Finder,
    screensaver — these never appear as apps and never count as time.

12. **Every AI surface uses the model you picked in Settings.** Chat, re-analyze,
    Apps Generate, briefs, weekly review — all use the same model. None secretly
    switches to a different one.

## How each view should work

### Timeline (the home screen)

Your day from midnight to now as a vertical list of blocks. Each block named for
what you did, sized by how long it took. A short recap on the right — generated by
the AI when the user clicks "Generate Recap," based on the blocks, apps, and sites
from that day. See `docs/specs/timeline.md` for the full spec.

### Apps (per-app breakdown)

Click an app and see what you did in it. Same intelligence as the Timeline, filtered
to one app. See `docs/specs/apps.md` for the full spec.

### AI (ask about your day)

The foundation benchmark: "What did I work on today, last week, last month?" — the
AI should answer confidently with evidence. It should be able to answer any question
about anything that was done on your laptop during that day, week, or month.

Think: "which video did I watch last week about aliens?" or "What does project
Daylens do?" or "how many hours did I spend in Cursor last week?"

**What's broken today:** Everything. It responds with "I don't have the tool
results… could you share the getDaySummary output again?" Walls of text instead of
tables. "Turn into bullets" refuses to work. Chat history vanishes when you switch
tabs. Wrong AI model used. The voice is apologetic, meta, and asks you to do the
app's job. See the screenshots.

**How it should work:**

- The app looks up your real data FIRST (blocks, times, threads), then hands it to
  the AI to phrase nicely. The AI never decides whether data exists — it narrates
  facts the app already fetched.
- "What did I work on today?" returns something like:

  > Today was an interesting one. You built on the progress of Wednesday and
  > continued with the timeline rework in Cursor and Claude Code, and you spent
  > some time on the Ghostty terminal working through networking configurations
  > for your work network in the evening.
  >
  > As a summary:
  > - Spent some time developing the timeline rework in Cursor and Claude Code
  >   from 8am to 10am
  > - Had class on ML pipeline from 10am to 1pm
  >
  > After lunch you spent some time on the Ghostty terminal working through
  > networking configurations until 9pm.
  >
  > And of course in between around 10am–10:40 watched some YouTube videos on AI
  > (I can see you're obsessed since even on Wednesday that's what you were doing
  > between 3pm and 4:40pm). Overall though, solid progress — keep it up.

- The voice is calm, confident, specific. Like the Dia morning brief. Never
  apologetic, never asks you to do its job.
- Use tables when they fit (time per activity, apps, sites). If the user asks for
  an exportable version in Excel format, do it.

See `docs/specs/ai.md` for the full spec.


### Morning brief

Two notifications:

1. **Yesterday's recap** — sent first thing when you open your laptop, ONLY IF you
   did not generate a recap the previous day. If you already generated one, this
   notification does not get sent. The notification itself should be readable
   without clicking — it needs to be written fresh every day with a real summary of
   the corresponding day.

2. **Carryover nudge** — sent after you've been working for 1–2 hours. Lists things
   you might have left open the previous day that you might want to pick up today.
   Also written fresh every day, relevant to that specific day.

**What's broken today:** A multi-slide carousel with focus scores.

**How it should work:**

- Greeting + carryover line ("The malaria notebook was still open — pick it up?" or
  "Nothing left open — clean start.") + "See yesterday" link.

Example morning brief tone:

> Good morning Tonny! Hope you had a great night's sleep. I can see you've been
> working on the timeline rework in Cursor and Claude Code from 8am to 10am —
> great work!
>
> Here's what you were doing yesterday that you'd want to pick up today:
> - The malaria Jupyter notebook was still open yesterday afternoon and you spent
>   a couple of hours on it.
> - Daylens v2 is getting closer to the PMF vision — great progress on the
>   timeline rework.

Then a second slide with interesting stats — time per activity, apps, sites used
that day and so far that week, with graphs and charts.

### Evening wrap

Same concept as the morning brief but for the end of the day. A notification sent
in the evening, written fresh and relevant to that specific day.

**What's broken today:** An 8-slide deck that lectures you about focus on a rest day.

**How it should work:**

- At most 5 cards: shape of the day, what you worked on, where time went, open
  thread (if any), quiet close.
- A leisure day collapses to 2 cards (shape + close) — and some funny guilt slides
  about where you could have been if you were more focused and productive.
- Totals and labels match the Timeline exactly.

### Weekly / monthly / annual wraps

Spotify Wrapped for your week, month, and year. What did you get done? What
matters? Told from the same data and the same tone as everything else.

**What's broken today:** Weekly totals disagree (20h7m vs 20h53m). "Main mode:
Entertainment" for a working developer. Monthly and annual don't exist yet.

**How it should work:**

- Built from frozen daily snapshots, not fresh summaries — so numbers always agree.
- Weekly review: chart with legend, totals that match, work-based "Main mode."

### Settings

**What's broken today:** AI model in Settings is ignored by re-analyze. Work
memory tags everything "browsing" at 65%. Label overrides don't propagate. MCP
on by default with dev paths.

**How it should work:**

- One AI model choice → used everywhere.
- Work memory learns varied, real categories with evidence and impact.
- Label overrides take effect across all views after recompute.
- MCP off by default in packaged builds.
- Every change that affects your data either triggers an update or says "applies
  to future activity only." The user should see the effect of their changes.

## How we build it

We fix the existing app. We don't add new features — we make what exists work as
expected. Be ruthless about removing legacy code and features that aren't needed.
If we're not adding things back, we haven't removed enough.

Rules for the build:

- Ship a feature → immediately clean up leftover code as the user gives feedback.
- No useless comments that don't add value.
- Plan in chat, not with piles of markdown files and planning code.
- Always clean up the codebase — avoid creating more docs, tests, and code that
  aren't needed.

**Future integrations** (not important right now — we have a lot to fix first):

- Calendar/email integrations
- Team/admin/cloud-sync features
- Mobile or browser extensions
- MCP feature expansion (just fix the settings bug)
- Visual redesign
