## 1. AI spend: $110/week, must become <$5/week — ROOT CAUSE FOUND

**Reported:** Usage page shows $110.21 in 7 days, 99.8% of it "Timeline labeling"
(76,063 calls · 82.7M tokens). Anthropic console CSV confirms: $221 over 30 days,
$209 of it Claude Haiku 4.5, $152 of that *uncached input tokens*.

**Root cause (verified in `ai_usage_events` in the live DB):**

- Every one of the ~77,000 background calls is `job_type = block_cleanup_relabel`,
  `trigger_source = background`, `screen = timeline_day`.
- The events cycle through the *same three blocks* (922 / 1027 / 1087 input tokens)
  every ~10 seconds, for hours — a relabel loop whose output never makes the block
  ineligible. `shouldReanalyzeBlockWithAI` keeps returning true when the AI's label
  fails `usefulBlockLabel` (weak/generic label), so the passive sweeps
  (`scheduleOvernightCleanup` queue + `runHistoryHeuristicUpgrade`'s `while (true)`)
  re-queue the same blocks forever. `listTimelineDaysNeedingHeuristicUpgrade` also
  never converges for a day that is processed but version-stale, because
  `getTimelineDayPayload` refuses to rematerialize processed days — the sweep's
  while-loop refetches the same dates endlessly.
- **The spender is the installed production app, not this tree.** The repo removed
  the passive path's callers and added the `DAYLENS_ENABLE_PASSIVE_AI` fuse on
  Jun 24, but `/Applications/Daylens.app` is **v1.0.44, built Jun 3** — it still
  contains the un-fused loop, it is a **login item (listed twice)**, and it shares
  the same `DaylensWindows` data dir and API key as the dev build. It has been
  burning ~$15–30/day in the background ever since. Calls stopped at 23:30 on
  Jul 4 — exactly when that process was quit and the dev app took over.

**Fix (code, this tree):**
- Delete the dead passive-AI path (cleanup queue, history heuristic upgrade,
  `scheduleTimelineAIJobs`) — it has no callers here; the once-per-day
  auto-analyze in `syncUploader.ts` (gated by `persistedDayWasProcessed`) replaced it.
- Add a hard **daily background-AI budget breaker** in `generateWorkBlockInsight`'s
  background path: count today's background `ai_usage_events`; past the cap, skip and
  log. Any future runaway loop caps at pennies instead of $30/day.

**Fix (founder action — the actual money switch):**
- Quit and replace `/Applications/Daylens.app` (v1.0.44) with a build of this tree,
  or delete it if the dev build is the daily driver. Remove the stale login item
  (Daylens appears twice in Login Items).
- Optional backstop: set a monthly spend limit on the key in the Anthropic console.

**Expected after fix:** legit spend is user-triggered Analyze + once-per-day
auto-analyze + chat/wraps ≈ 50–100 small Haiku calls/day → well under $5/week.

## 2. Activity colors inconsistent across days — stale categories, not colors

**Reported:** the right activity color only appears after clicking Analyze, or on
days since the feature shipped; older days don't get colors.

**Root cause:** the renderer already colors every block from `dominantCategory`
(`Timeline.tsx` `CalendarBlockCard`; the provisional-grey override was removed
2026-07-03). The live DB shows every persisted block *has* a category — but old
days carry categories computed by superseded heuristics (`timeline-v7`/`v8` vs
current) — e.g. Jul 4 persisted 5 of 7 blocks as `entertainment` (dim slate ≈
"no color"). `getTimelineDayPayload` returns persisted blocks as-is for processed
days (`persistedDayWasProcessed` → early return), so stale categories are frozen
forever; only a manual Analyze rebuilds them. The color bug is the category bug.

**Fix:** when a persisted day's `heuristic_version` is stale, recompute the
deterministic block facts (category, distribution, evidence) on read while
preserving labels and user corrections (invariant 8) — no AI required, so it's
free and instant. Colors then converge on every historical day.

## 3. Idle-gap spacing renders inconsistently

**Reported:** the space between idle time and the next block is sometimes too
short; idle time gets blocked/barely visible, especially while a day is going on.

**Root cause:** `Timeline.tsx` `CalendarDayTrack` floors every card's *painted*
height at `MIN_DAY_CARD_HEIGHT` px while keeping its top at the true clock
position — a short block paints past its real end time, covering the idle gap
(and its "Idle · 12m" caption) below it. Height = duration (invariant 4) is
violated exactly when a block is short.

**Fix:** keep the readability floor, but clamp a floored card so it never paints
past the next block's top (or the gap label zone) — proportional layout wins
whenever the floor would cover real empty time.

## 4. Apps view: unify "What you did there" / "Time by domain" / "Pages visited"

**Reported:** for browser apps these three sections should converge into one,
and the math must reconcile exactly: 1h56m in Dia = Σ domains = Σ pages.

**Status:** ✅ implemented (same session) with the recommended defaults; the
three founder questions below were decided as: (1) keep "What you did there"
as the lead-in, (2) foreground-only totals, (3) domains collapsed by default —
flag it if any of those should flip.

**Root cause of the disagreement (verified):** the three sections use three code
paths on two time bases — the header total is foreground app_session seconds
(`getAppSummariesForRange`), while "Time by domain" and "Pages visited" are raw
`SUM(duration_sec)` over browser history (`getDomainSummariesForBrowser` /
`getPageSummariesForBrowser`), which keeps accruing while the browser is in the
background and double-counts overlapping visits. Commit 4d140aa already built
the fix for the timeline paths (`reconcileWebsiteVisits`: clip visits to
frontmost slices, disjoint credit — Σ pages = domain by construction) and
explicitly left App Detail as the known remaining raw-SUM path.

**Plan:**
- One new query `getBrowserActivityBreakdown(...)` built on
  `reconcileWebsiteVisits`, keyed on `canonical_browser_id`, clipped to the
  browser's own foreground intervals. Delete `getDomainSummariesForBrowser`,
  `getPageSummariesForBrowser`, and `topDomainsForBrowser`.
- One payload shape: `browserActivity = { totalSeconds (== header), domains:
  [{domain, totalSeconds, pages: [...] }], unattributedSeconds }` — Σ pages =
  domain and Σ domains + unattributed = header, by construction. The remainder
  renders as an explicit non-interactive "No page recorded · Xm" row
  (invariant 10 — never smear the gap into a domain).
- UI: one "Where your time went" section; domain rows expandable into page rows
  (existing open/delete wiring reused); "What you did there" stays as the
  intent-level lead-in above it (different axis — it doesn't reconcile to
  domain seconds and shouldn't pretend to).
- Tests: exact reconciliation on a seeded DB (background accrual + double
  capture + capture gap), honest remainder, non-browser degrade.

**Founder questions:**
1. Keep "What you did there" as the lead-in above the merged section
   (recommended), or drop it for browsers entirely?
2. Browsing during honest capture gaps: header is foreground-only, so the
   merged view excludes it. Acceptable? (Counting it would desync the Apps rail.)
3. Domains all collapsed by default (recommended), or top domain auto-expanded?

## 5. AI tab: input box + provider error copy

**Reported:** (a) the composer is oversized and text isn't centered; it should
start compact and grow with content (Cursor-style reference). (b) Expired key /
exhausted tokens show the generic "Couldn't complete that" instead of the real
cause.

**Root cause:** (a) `AICompose.tsx` styling. (b) `providerErrors.ts` already
classifies auth/credit/quota/rate-limit errors with good copy — but the chat
path reaches the `unknown` fallback, meaning the raw provider error loses its
shape (status/type) before classification. Trace and classify at the throw site.

**Fix:** compact auto-growing composer; ensure the original SDK error object
reaches `classifyProviderError` unwrapped so auth/credit/quota render their real
messages.
