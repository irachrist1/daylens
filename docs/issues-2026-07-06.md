## Open items from the tracking repair

- **Dia private windows are not structurally detectable.** Verified live: Dia's
  AppleScript dictionary exposes no `mode of front window` (Chrome does). A Dia private
  window without a title marker still can't be recognized. The fix at least stops the
  history-DB fallback from misattributing private dwell to regular sites. Revisit when
  Dia ships a scripting property; consider filing with the Browser Company.
- **Historical processed days keep their (sometimes wrong) boundaries.** Days built
  before the sleep fix can contain blocks that span overnight holes (e.g. Jul 4: 12.3h
  of raw sessions inside 22.1h of block span). v10 recolors them but — by the frozen-
  boundaries policy — does not re-split them. Right-click → re-analyze any day whose
  shape looks wrong; a bulk rebuild was deliberately not run because it would discard
  AI labels on real days without the founder's say-so.
- **Screen Time parity:** after the fix, per-app totals are bounded by real input
  activity, but Screen Time counts differently (per-display, includes some background
  media). Expect same-order agreement, not equality. Re-check after one full day on the
  fixed build.

## Full-codebase sweep — verified findings beyond the reported bugs

Ranked. Each was verified in the implementation; file references are anchors, not
guesses.

### 1. HIGH (security) — auto-updater installs unverified bundles
`src/main/services/updater.ts` (`downloadRemoteInstaller`, `scheduleAdhocMacSwap`):
the macOS ad-hoc update path downloads a ZIP from the feed's `installUrl`, re-signs it
ad-hoc, strips the Gatekeeper quarantine (`xattr -cr`), and launches it. The ONLY
validation is a byte-count comparison; `RemoteUpdateDescriptor`
(`src/shared/updaterReleaseFeed.ts`) has no hash or signature field. Anyone who can
influence the feed or the download host serves a same-size malicious app that Daylens
itself un-quarantines — silent code execution. **Fix direction:** add `sha256` (ideally
also a minisign/Ed25519 signature) to the descriptor, verify before staging, never
strip quarantine on unverified bytes. No test exercises the download/verify/swap path.

### 2. HIGH (perf) — per-poll synchronous full browser-history copies
`src/main/services/browserContext.ts` (`copyHistoryDb` → `fs.copyFileSync` of the whole
History DB + WAL + SHM), reached from `recentHistoryTab` on every tab read that misses
the AppleScript path (all reads on Linux, fallback on macOS/Windows). `poll()` contains
no `await` — each copy blocks the Electron main process for the duration of a
multi-tens-of-MB file copy, up to every ~30s of browsing. This is the residual cause of
poll-tick stutter (F19 was a partial mitigation). **Fix direction:** open the source DB
read-only immutable, or copy only when mtime/size changed, or move to a worker thread.

### 3. MEDIUM (data integrity) — `insertAppSession` cannot signal a dedup drop
`src/main/db/queries.ts` (`insertAppSession`): `INSERT OR IGNORE` +
`return result.lastInsertRowid` unconditionally — on a `(bundle_id, start_time)`
conflict it returns a stale rowid (some other row's id) instead of null, so callers
treat a dropped insert as a success and id-keyed follow-ups mis-associate. The sibling
`insertWebsiteVisit` already does this right (`result.changes > 0`). **Fix direction:**
`return result.changes > 0 ? Number(result.lastInsertRowid) : null`.

### 4. LOW (AI quality) — sanitizer backstop over-redacts
`src/shared/aiSanitize.ts` generic token patterns (`hex_blob`, `base64_blob`,
`generic_token` ≥32 chars) strip legitimate long identifiers from model input. Safe
direction (over-redaction, never a leak) but degrades answers about hashes/ids.
**Fix direction:** entropy-gate the generic backstop.

### Checked and explicitly CLEAR (so nobody re-audits them from stale docs)
- **MCP server**: stdio transport only, DB opened read-only — no network exposure.
- **Electron hardening**: `contextIsolation: true`, `nodeIntegration: false`, no
  `webSecurity` opt-outs, no remote-debugging in production config.
- **Invariant 12** (every AI surface uses the Settings model): holds in
  `aiOrchestration.ts`; the historic cross-provider bug is fixed.
- **Migrations**: each migration + version bump runs in one transaction.
- **Dead capture tables** (`idle_periods`, `raw_window_sessions`,
  `browser_context_events`, `file_activity_events`): already dropped by migration v42 —
  code is ahead of the docs here. (Note: `idle_periods` still exists in the founder's
  live DB with 0 rows; it predates v42's drop on that DB — harmless.)
- **Interval timers / analytics PII / localDate DST arithmetic**: no issues found.
